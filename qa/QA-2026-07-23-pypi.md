# QA Report: pypi.org

**Tier:** Quick
**Date:** 2026-07-23
**Tester:** Hermes + qa skill methodology

## Verdict
✅ **PASS**

## What works
- **Landing page** (`/project/requests/`) loads with HTTP 200
- Latest version `2.34.2` clearly displayed
- "Copy PIP instructions" button works (renders the pip command)
- **3 tabs** (description / history / download files) all navigate
- **Release history** tab shows the last 50+ releases with proper
  timestamps in `YYYY-MM-DD HH:MM:SS (+TZ)` format
- **GitHub Statistics** clearly show: 54,146 stars, 10,028 forks,
  146 open issues, 87 open PRs
- **Maintainers** section lists 3 maintainers with avatars
- **Verified vs Unverified details** are properly separated
- License, author, requires-Python all visible

## Bugs found
None at critical/high severity.

### Medium (notable but not failing)
- **Search triggers CAPTCHA for headless browser sessions** — typing
  "requests" in the search bar and clicking "Search" landed on
  `/search/?q=requests` which then served a CAPTCHA challenge. This
  is not a bug in production browsers (real users get results) but it
  will trip up any automated QA tooling. PyPI uses Fastly + custom
  CAPTCHA. Workaround: use the JSON API directly.
  - Repro: from a headless browser, navigate `/` → type query → submit
  - Severity: medium (only impacts automation)

### Low
- Release-history timestamps show `(+07:00)` consistently, but the
  `Released:` paragraph on the landing page just shows "May 15, 2026"
  without time-of-day. Minor inconsistency, but acceptable for a
  human-facing summary.
- The "stars/forks/issues/PRs" widget lists GitHub stats even though
  the project may have moved to GitLab (not applicable to `requests`,
  which is on GH). Not necessarily wrong, but a stale assumption.

## Notes
