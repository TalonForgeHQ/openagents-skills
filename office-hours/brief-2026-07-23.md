# Brief: OpenAgents Skills — vendor-neutral skill suite

## The user
Zinou (zinou potts), indie dev / AI agent operator on Windows. Built
`atlas-store` — a revenue-loop system for AI-agent productized services.
Already runs Hermes Agent as his daily driver. Wants audience + open-source
credibility. Single user, not a team — every skill he adopts becomes a
default he lives with.

## The wedge
A vendor-neutral skill format (canonical tool names + Markdown frontmatter)
so the same `SKILL.md` works in Hermes Agent, Claude Code, Codex CLI,
Cursor, Aider, and bare LLMs. The closest competitor — garrytan/gstack
(123k stars) — is locked to Claude Code's runtime. We are not.

## Why now
Three forces converged this week:
1. Garry Tan's gstack hit escape velocity (123k stars) but has no escape
   hatch out of Claude Code.
2. Multi-agent workflows are now real (Claude Code + Codex CLI + Hermes
   + Cursor all shipping skills/plugins in 2026) — vendor lock-in is
   the obvious next pain point.
3. The audience for "agent-agnostic tooling" is the same audience that's
   about to discover they need it: indie devs running multiple agents.

## The 10x moment
A new skill can be written once and inherited by every agent on the
user's box — instead of being forked per-agent. Specifically: when a
"spec this out" trigger works the same in Hermes, Claude Code, and
Codex, the user feels like the underlying capability *transcends* the
agent. That's the platform feel without being a platform.

## What kills this
- Garry Tan adopts our format upstream (best outcome, our repo becomes
  a staging area), or he ignores us and the gstack fanbase never hears
  about us (likely). We need a non-Garry distribution channel.
- The cross-agent mappings turn out to be too lossy — Claude Code
  users find our stubs don't actually do what gstack did, and we get
  a "this is a downgrade" reputation.
- A bigger player (Anthropic, OpenAI, Vercel) ships a cross-agent
  skill format first and we look derivative.

## The cheap test
Pick the 5 highest-leverage skills (office-hours, spec, investigate, qa,
scrape). Write them as real production bodies — not stubs. Run them in
Hermes against real tasks this week. Count: how many times did the
agent's output improve because the skill was loaded? If the answer is
"<2 of 5", the methodology is shallow. If it's ">=4 of 5", we have a
shippable asset.

## 30-day ship target
One Show HN post about a working OpenAgents Skills repo + 11 real
production skills + 25 honest stubs + the install scripts. Public
repo at github.com/TalonForgeHQ/openagents-skills, default branch
`main`, single-commit history, AUDIT.md done. The production skills
must actually work when invoked — not "might work, see upstream".
